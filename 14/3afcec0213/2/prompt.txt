# /start-review — Triple-Layer Code Review Team

**PR**: 274

You are setting up a triple-layer code review for PR `274`. This can be used standalone or as part of a `/start-feature` team workflow. The review loops until all FIX NOW items are resolved — it does NOT end after a single pass.

---

## Step 1: Get PR Details

Fetch the PR details:

```bash
# Get the correct owner from git remote
git remote get-url origin | sed 's/.*[:/]\([^/]*\)\/[^/]*\.git/\1/'
```

Use `gh pr view 274` to get PR title, description, and diff stats.
Use `gh pr diff 274` to get the diff.

Determine the PR scope:
- Count files changed and lines changed per file
- Identify if it touches schema concerns (types.ts, SQLite schema, server API)
- Classify changed files into domains for specialist selection (Step 2)

---

## Step 2: Select Domain Specialists

Instead of fixed generalist reviewers, select 1-2 **domain specialists** based on what the PR actually changes. Each specialist combines deep domain expertise with general engineering awareness. See `docs/REVIEW-SPECIALISTS.md` for the full specialist registry and prompt templates.

### Domain Classification

Classify each changed file into a domain based on file patterns:

| Domain | File Pattern Triggers |
|--------|----------------------|
| **SQL/Database** | `server/src/routes/*.ts` (when containing SQL keywords), `cli/src/db/*.ts` |
| **React/Frontend** | `dashboard/src/**/*.tsx`, `dashboard/src/**/*.ts` |
| **Node/CLI** | `cli/src/commands/*.ts`, `cli/src/utils/*.ts`, `server/src/*.ts` (non-route, non-LLM) |
| **Parser/Provider** | `cli/src/providers/**/*.ts`, `cli/src/parser/*.ts` |
| **LLM** | `server/src/llm/**/*.ts` — handled by LLM Expert (separate, unchanged) |

Files that don't match any domain (e.g., `package.json`, `tsconfig.json`, `.gitignore`, `CLAUDE.md`) are ignored during classification.

### Selection Algorithm

1. For each domain, sum the lines changed across all files in that domain
2. **Primary specialist** = domain with the most lines changed
3. **Secondary specialist** = next domain IF it has >= 30% of total changed lines AND is a different domain
4. Cap at **2 domain specialists** (LLM Expert is a separate conditional reviewer)
5. If a PR changes ONLY non-domain files, use the **Node/CLI Specialist** as general-purpose

### LLM Expert (Conditional — Unchanged)

**Invoke LLM Expert if ANY of these apply:**
- PR touches `server/src/llm/` (prompts, providers, export-prompts)
- PR adds or modifies LLM API calls
- PR changes structured output schemas or SSE streaming
- PR modifies token budgets or model selection logic
- PR adds new LLM-powered features

**Skip LLM Expert if ALL of these apply:**
- No LLM code touched
- Pure UI, CLI, or schema changes
- Provider (source tool) implementations only

### CLI UX

### Issues Found

**🟡 SUGGESTION — "sessions" label may be misleading in up-to-date message**

At `cli/src/commands/sync.ts:158` (post-diff line), the new message is:
```ts
log(chalk.gray(`  ✔ Up to date (${sessionFiles.length} sessions)`));
```

`sessionFiles` is the result of `provider.discover()`, which returns **file paths** (one JSONL file per session for claude-code, but the semantics may differ for other providers like cursor which uses a single `state.vscdb`). For claude-code this is accurate (1 file = 1 session), but for cursor, `sessionFiles.length` could be `1` even when there are many sessions inside that single vscdb file.

Looking at the PR screenshot, cursor shows `Up to date (170 sessions)` — this is likely the actual session count from the database, not the file count. So either:
- The cursor provider's `discover()` returns session-level entries (making the label correct), OR
- The number shown is coincidental/from a different provider

Worth a quick verification that `sessionFiles.length` semantically equals "number of sessions" across all 5 providers. If it doesn't, consider labeling it `(${sessionFiles.length} files)` or fetching the actual session count from the DB for the up-to-date message.

**🟡 SUGGESTION — `spinner.stop()` leaves no visual trace for usage stats**

At `cli/src/commands/sync.ts:256-257` (post-diff):
```ts
recalculateUsageStats();
spinner.stop();
```

`spinner.stop()` clears the spinner line entirely, which means the spinner text `Recalculating usage stats...` flashes and vanishes. This is fine if the operation is fast (<100ms), but if it takes noticeable time on large databases, the user sees a spinner appear and vanish with no explanation. The current approach is acceptable for the stated goal (suppress internal housekeeping), but if the operation ever becomes slow, consider `spinner.stop()` before starting rather than starting and stopping.

**🔵 NOTE — Telemetry notice lost the "Details" pointer**

The old banner included `Details: code-insights telemetry status`. The new condensed line only mentions the disable command. Users who want to see what's collected now have no pointer to `telemetry status`. This is a minor discoverability loss. Could be:
```
Telemetry enabled · Details: code-insights telemetry · Disable: code-insights telemetry disable
```
But this may be intentionally trimmed for brevity — author's call.

**🔵 NOTE — `chalk` import added to telemetry.ts**

The new `import chalk from 'chalk'` in `cli/src/utils/telemetry.ts` is fine — chalk is already a dependency used throughout the CLI. No issue, just noting for completeness.
