## Pretext Evaluation: Not a Fit for Our CLI

The UX engineer did a thorough audit of both the `pretext` library and our entire CLI output codebase. Here's the verdict:

### What pretext is

A **browser-only** (v0.0.3) library for measuring multiline text layout **without triggering DOM reflows** — it uses `canvas.measureText()` and pure arithmetic to compute text height and line breaks in proportional fonts.

### Why it doesn't apply to us

**Fundamental runtime mismatch.** Our CLI outputs to a terminal, which is a completely different world:

| Browser (pretext's domain) | Terminal (our domain) |
|---|---|
| Variable-width proportional fonts | Monospace character grid |
| Pixel-based measurement | Column-based (`process.stdout.columns`) |
| DOM reflows are expensive | No layout engine — characters go to stdout |
| Needs `canvas.measureText()` API | Needs `string.length` (or `string-width` for CJK) |

The problem pretext solves — avoiding layout reflows for variable-width text — **does not exist in a terminal**.

### Our current approach is already solid

The agent reviewed all our render utilities (`cli/src/commands/stats/render/`) and found:
- Responsive breakpoints at 4 terminal widths (60/80/100+)
- Dynamic bar charts, metric grids, sparklines
- Proper `process.stdout.columns` handling with fallback to 80

### What would actually help (no library needed)

If we want to polish our terminal output, these are the real opportunities — each 10-30 lines of code:

1. **Smarter label truncation** — bar chart labels are hard-coded at 20 chars. Could use proportional available space instead.
2. **CJK-aware column counting** — `string-width` (already a transitive dep via ora/chalk) for accurate widths with non-Latin text.
3. **Text wrapping for reflect narratives** — wrap at terminal width instead of relying on terminal auto-wrap.

### Recommendation

**Skip adoption.** Pretext is impressive engineering for browser text layout, but it requires browser APIs (`canvas`, `Intl.Segmenter`), is pre-1.0 (v0.0.3), and solves a problem that doesn't exist in our Node.js terminal environment.

Want me to pursue any of those three small CLI polish improvements instead? They'd deliver real UX value with minimal code.

## Dashboard Evaluation: Also a Skip (for now)

The runtime mismatch is gone — our dashboard **is** a browser SPA. But the UX engineer found that **we don't have the problem pretext solves**.

### What pretext excels at

Pre-computing text height/line breaks **without DOM reflows** — primarily for virtualized lists and layout-shift prevention.

### Why our dashboard doesn't need it

| Pretext strength | Our dashboard reality |
|---|---|
| Virtualized list height prediction | No virtualization — session list caps at 200, messages render flat |
| Avoiding layout shift | Local-first, loads from localhost — no network-driven CLS |
| Pixel-accurate text truncation | CSS `truncate`, `line-clamp-1/2/3` handles everything |
| Shrink-wrap bubble sizing | Chat bubbles use `max-w-[80%]` — CSS constraint is sufficient |

### The only two places doing text measurement today

1. **`share-card-utils.ts`** — already uses `canvas.measureText()` directly in a 7-line loop for PNG export. Works correctly. Replacing with pretext = same result, new dependency.
2. **`AnalyticsPage.tsx`** — project name truncation via `slice(0, 15) + '...'`. Character-based, not pixel-perfect, but perfectly readable in practice.

### Additional concerns

- **v0.0.3** — pre-1.0, API still changing, active research phase
- **~15-25KB** of bidi, CJK, emoji handling code we'd never exercise
- **Requires pinned font names** — doesn't support `system-ui`, which our dashboard uses
- **No React integration** — we'd build our own hooks around it

### When to reconsider

- If we add **conversation virtualization** (500+ message sessions)
- If we build a **canvas-rendered timeline** or heatmap with text labels
- When pretext reaches **1.0 with a stable API**

### Verdict

**Skip.** Cool library, wrong timing. CSS handles our text layout today, and we don't have virtualization or layout-shift problems that justify a pre-1.0 dependency. The project's principle applies here too — we don't add infrastructure without a product need driving it.
