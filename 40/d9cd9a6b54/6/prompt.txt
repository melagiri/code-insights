## Key Design Decisions (Agreed)

### 1. Flag Naming: `--native` (not `--inline`)

The cofounder recommends **`--native`** over `--inline`:
> "Native" is self-documenting in developer ecosystems — it means "using the platform's own capabilities." `--inline` suggests "in the same process," which is misleading since we shell out to `claude -p`.

### 2. `--hook` Flag for stdin Parsing

The TA proposes a clean addition — when called from a hook, the CLI reads session context from stdin:

```bash
# Hook invocation (clean, no shell escaping):
code-insights insights --hook --native -q

# Manual invocation:
code-insights insights <session_id> --source claude-code

# Manual with native (if user has claude CLI):
code-insights insights <session_id> --native
```

When `--hook` is passed, the command reads JSON from stdin (`session_id`, `transcript_path`) instead of requiring positional args.

### 3. Session Resume: Upsert, Latest Wins (Option A)

Both agents agree — **don't overthink the resume problem**:
- Track `session_message_count` in `analysis_usage` (new nullable column, V8 migration)
- On re-invocation: if message count matches → skip. If changed → re-analyze
- DELETE + INSERT in a transaction (existing pattern, no new code)
- Never surface "preliminary" vs "final" to users — just silently update

### 4. Sync-Before-Analyze via `syncSingleFile()`

The TA identified the real race condition and proposed a targeted fix:

```
SessionEnd hook → code-insights insights --hook --native -q
                      │
                      ├── 1. syncSingleFile(transcript_path)  ← 50-200ms
                      ├── 2. claude -p (analysis)              ← 5-15s
                      ├── 3. claude -p (prompt quality)        ← 5-15s
                      └── 4. Save to SQLite                    ← ~10ms
                                                        Total: ~10-30s
```

This avoids a full sync (which scans all providers/files) and guarantees fresh data.

### 5. Prompt Sharing: Move to CLI, Server Re-exports

The TA recommends moving prompt builders from `server/src/llm/` → `cli/src/analysis/`, with server re-exporting. This:
- Eliminates circular dependency (server already depends on CLI)
- Keeps single source of truth
- CLI can build prompts without starting the server

```
cli/src/analysis/
  ├── prompts.ts              ← moved from server
  ├── prompt-types.ts
  ├── prompt-constants.ts
  ├── message-format.ts
  ├── response-parsers.ts
  └── schemas/
      ├── session-analysis.json   ← hand-maintained, test-validated
      └── prompt-quality.json
```

### 6. JSON Schemas: Hand-Maintained + Test-Validated

Hand-maintained JSON schema files (not auto-generated) because:
- `claude -p --json-schema` needs flat schemas (no `$ref` chains)
- Full control over shape
- Sync enforced by test that compares schema properties against TypeScript types

`★ Insight ─────────────────────────────────────`
- **The dependency inversion is the cleverest part of this design.** Prompts move to CLI (the leaf node), server re-exports from CLI. This follows the "dependency rule" — business logic (prompts) lives in the innermost layer, infrastructure (server, hooks) wraps around it.

- **The cofounder's positioning is right: "Zero-config for Claude Code users"** — not tied to any subscription tier. Every Claude Code user (Pro, Max, Team, Enterprise) benefits equally.
