### Findings

#### 🟡 Suggestion: Stale comment in `facets.ts` (on-disk, not in diff)

The comment block at lines 120-121 of `server/src/routes/facets.ts` describes the outdated query as checking two conditions:
```
// Returns count of session_facets rows where:
//   - effective_patterns entries lack a category field, OR
//   - friction_points entries lack an attribution field
```

The PR diff adds a third line to the comment (`effective_patterns entries lack a driver field`), but looking at the diff hunk, the comment update is applied to the NEW file correctly. However, the hunk header shows the old comment at line 118-121 is being replaced — confirmed this is fine. No issue here after closer inspection.

#### 🟡 Suggestion: `getDominantDriver` mutation of input via `.sort()`

In `PatternsPage.tsx`, `getDominantDriver` calls `entries.sort(...)` which mutates the array returned by `Object.entries()`. While `Object.entries()` creates a new array (so the original object is safe), it would be slightly cleaner to use `reduce` or `toSorted()` to find the max without sorting. This is minor — no actual bug since the entries array is local to the function.

#### 🟡 Suggestion: Driver value validation is absent

In `shared-aggregation.ts`, the driver value from JSON is accumulated into the `drivers` record without validating it's one of the three expected values (`user-driven`, `ai-driven`, `collaborative`). If the LLM returns an unexpected driver string (e.g., a typo like `"user_driven"` or a novel value), it would be stored and aggregated as-is, and the dashboard would render it with the fallback style (`bg-muted text-muted-foreground`) and raw string label.

This matches how `category` is handled (normalize at write time, belt-and-suspenders at read time), so it's consistent. But unlike categories, there's no driver normalizer. Consider whether a simple validation/normalization (e.g., mapping `"user_driven"` → `"user-driven"`) would be worthwhile, or if the observability warning in `parseAnalysisResponse` is sufficient.

#### 🟢 Note: SQL parameter spreading is correct

The outdated query now has three UNION arms, each using `${where}` which consumes `...params`. The `.get(...params, ...params, ...params)` correctly spreads params three times. This pattern is consistent with how the two-arm version worked before.

### YELLOW — Suggestions: Should consider

**1. Outdated query performance with 3 UNION arms and repeated params**

The `/api/facets/outdated` query now has 3 UNION subqueries, each doing a CROSS JOIN with `json_each()`. The params are spread 3 times (`...params, ...params, ...params`). This works correctly, but:

- Each arm does a full CROSS JOIN + json_extract scan. For a user with 500+ sessions, this is 3 separate table scans with JSON parsing.
- The comment says "UNION deduplicates" but each arm already does `SELECT DISTINCT` — the DISTINCT inside each arm is redundant when UNION (not UNION ALL) already deduplicates across arms.

**Suggestion:** Remove `DISTINCT` from individual arms (UNION handles it), or consider combining the effective_patterns checks into a single arm with `OR` since they share the same table join:

```sql
-- Instead of two separate effective_patterns arms:
SELECT sf.session_id
FROM session_facets sf
JOIN sessions s ON sf.session_id = s.id
CROSS JOIN json_each(sf.effective_patterns) je
WHERE ...
AND json_array_length(sf.effective_patterns) > 0
AND (json_extract(je.value, '$.category') IS NULL
     OR json_extract(je.value, '$.driver') IS NULL)
```

This halves the effective_patterns scanning work and reduces param spreading from 3x to 2x.

**2. `getDominantDriver` tie-breaking is arbitrary**

When two drivers are tied (e.g., `{ "user-driven": 3, "ai-driven": 3 }`), `Array.sort` is not stable across all engines, so the displayed pill is non-deterministic. With small sample sizes (e.g., a pattern seen 4 times), ties are common.

**Suggestion:** Add a deterministic tiebreaker. Options:
- Alphabetical fallback (simple)
- Priority order: user-driven > collaborative > ai-driven (semantically meaningful — when in doubt, credit the user)
- Show "Mixed" when no driver exceeds 50% (more honest)

**Suggestion:** Consider a tooltip showing the full breakdown on hover (e.g., "User: 5, AI: 4, Collab: 3"). This keeps the UI clean while providing transparency. Not blocking — the pill is useful as-is, but the tooltip would be a low-effort improvement.

**Suggestion:** Pick one concrete value for the example (e.g., `"driver": "collaborative"`) and add a comment like `// one of: user-driven, ai-driven, collaborative`. The enum constraint is already clearly documented in the classification guidance above — the example should show a valid value.
