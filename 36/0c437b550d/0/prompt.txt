**1. `successRate` computed but unused in `WeekAtAGlanceStrip`**

const successRate = outcomeTotal > 0 ? Math.round((successCount / outcomeTotal) * 100) : 0;

This variable is computed in the parent component but never used there — `WorkingStyleShareCard` computes its own `successRate` internally. This is dead code that should be removed. Not a bug, but it signals copy-paste drift that could cause confusion.

**Fix:** Use a ref for the download-in-progress guard instead of reading state in the callback, or use `disabled` on the button as the sole guard (which the code already does — so the `isDownloading` check in the callback is redundant but harmless). **Severity: Low-blocking** — the button `disabled` prop prevents real user double-clicks, but the pattern is fragile.

**3. Off-screen card stays mounted in the DOM permanently**
The `WorkingStyleShareCard` is rendered with `position: absolute; left: -9999px` and stays in the DOM whenever `hasGenerated && displayTagline` is true — which is the entire time the user is on the Patterns page after generating. This is a hidden 1200x630 div with children.

### 2. `CHARACTER_DISPLAY_NAMES` duplicates `SESSION_CHARACTER_LABELS` from `constants/colors.ts`

`WorkingStyleShareCard.tsx` defines its own `CHARACTER_DISPLAY_NAMES` map:
```ts
const CHARACTER_DISPLAY_NAMES: Record<string, string> = {
  deep_focus: 'Deep Focus',
  bug_hunt: 'Bug Hunt',
  ...
};
```

This is an exact copy of `SESSION_CHARACTER_LABELS` from `dashboard/src/lib/constants/colors.ts` (lines 44-52). If a new character type is added to the canonical map, this copy will silently go stale.

**Recommendation:** Import `SESSION_CHARACTER_LABELS` and use it. The "no CSS variables" constraint applies to colors, not to string labels. A comment like `// Safe to import — pure strings, no CSS` would clarify the intent.

### 3. `SOURCE_TOOL_DISPLAY_NAMES` is a third copy of tool display names

The CLAUDE.md documents 5 source tools. The share card utils define their own display name map. There may already be display name maps in the CLI or server. Every time a new source tool is added (the project already went from 1 to 5), three+ maps need updating.

**Recommendation:** At minimum, add a `// keep in sync with` comment pointing to the canonical list. Ideally, share a single map if the dashboard already has one.

### 4. Milestone thresholds — streak uses "Day" but WeekAtAGlanceStrip shows "w" (weeks)

The streak milestone labels say `"7-Day Streak"`, `"14-Day Streak"`, etc. But in the main UI (`WeekAtAGlanceStrip`), streak is displayed as `{streak}w streak` — using "w" for weeks. This inconsistency means the same number is labeled as days in the share card and weeks in the dashboard.

**Which is correct?** If the streak value is "consecutive weeks with sessions" (which the ISO week navigation suggests), then the share card labels are wrong. If it's truly days, then the dashboard abbreviation `w` is misleading.

**Recommendation:** Verify the streak unit and make both consistent.

### 5. `handleDownload` has `isDownloading` in its `useCallback` dependency but reads it as a guard

```ts
const handleDownload = useCallback(async () => {
    if (!shareCardRef.current || isDownloading) return;
    setIsDownloading(true);
    ...
}, [isDownloading]);
```

Every time `isDownloading` changes, the callback is recreated. The `isDownloading` guard is to prevent double-clicks, but `disabled={isDownloading}` on the button already handles that. The guard is redundant. You could simplify to `useCallback(async () => { ... }, [])` with just the `disabled` prop for protection, or use a ref for the guard to avoid the dependency.

This is minor but the current pattern means the callback identity changes on every download start/finish, which is unnecessary.

### 6. The off-screen card is always mounted in the DOM when tagline exists

```tsx
{hasGenerated && displayTagline && (
    <WorkingStyleShareCard ref={shareCardRef} ... />
)}
```

This means the 1200x630 card with SVGs, gradients, and multiple child elements is rendered into the DOM on every visit to the Patterns page (after generation), even if the user never clicks Share. It's lightweight enough that this is probably fine, but it's worth noting that a lazy approach (mount on click, capture, unmount) would be cleaner and avoid any potential layout impact from the off-screen positioned element.

**Not blocking** — the off-screen positioning works, and `html-to-image` needs a mounted element. Just a design tradeoff to be aware of.

### 7. `getMonthYear()` uses runtime date, not the selected week

The footer says `"Patterns . Mar 2026"` using `new Date()`. But the Patterns page uses ISO week navigation — the user could be viewing `2026-W08` (February) while the card says "Mar 2026". The month should reflect the selected week's date range, not today.

**Recommendation:** Pass the selected week (or its date range) to the share card and derive the month from it.

# /release — Automated Release Workflow

**Arguments**: minor

You are executing the release workflow for `@code-insights/cli`. Parse `minor` to extract:
- **type** (required): `patch`, `minor`, or `major`
- **description** (optional): A one-liner for the release title

If type is missing or not one of `patch`/`minor`/`major`, ask the user to provide it.

## Step 1: Pre-flight Checks

Run ALL of these checks. If any fail, STOP and tell the user what to fix.

```bash
# Must be on master
git branch --show-current  # Must output "master"

# Must have clean working tree
git status --porcelain     # Must be empty

# Pull latest
git pull origin master
```

Read the current version from `cli/package.json` (the `"version"` field).

Compute the new version:
- `patch`: increment the third number (3.6.1 → 3.6.2)
- `minor`: increment the second number, reset third to 0 (3.6.1 → 3.7.0)
- `major`: increment the first number, reset second and third to 0 (3.6.1 → 4.0.0)

**Resume detection**: Run `npm view @code-insights/cli version` to get the currently published version. If the version in `package.json` is ALREADY GREATER than the npm version, this is a **resumed release** — skip to Step 6.

---

## Step 2: Analyze Changes

Find the last git tag and analyze what changed:

```bash
# Find latest tag
git describe --tags --abbrev=0

# Commits since last tag
git log $(git describe --tags --abbrev=0)..HEAD --oneline

# Full diff since last tag
git diff $(git describe --tags --abbrev=0)..HEAD --stat
```

Read the full diff and commit messages. Then:

1. **Generate a CHANGELOG entry** in the existing format (see `cli/CHANGELOG.md` for style). Group changes under `### Added`, `### Changed`, `### Fixed`, `### Improved` as appropriate. Match the existing tone — concise but descriptive, with bold lead-ins.

2. **Generate a release title** — a short phrase summarizing the release (e.g., "LLM-Powered Export Page", "PostHog custom domain"). Use the user's description if provided, otherwise derive from the changes.

3. **For minor/major releases**: Assess which docs need updating:
   - `docs/ROADMAP.md` — if new milestones were completed
   - `docs/PRODUCT.md` — if product capabilities changed
   - `docs/VISION.md` — for major releases, if the vision expanded
   - `README.md` — if user-facing features or setup changed
