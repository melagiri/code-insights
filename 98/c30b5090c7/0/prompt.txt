### BLOCKING

**1. Unbounded week generation can hang or OOM on corrupt data**
`server/src/routes/reflect.ts` — the `while (weekMondayMs >= earliestMondayMs - 86400000)` loop

If `earliestRow.earliest` contains a corrupt or very old timestamp (e.g., epoch `1970-01-01` from a bad session import), this loop generates ~2,900 weeks (56 years). Each iteration calls `formatIsoWeek` + `parseIsoWeek` + allocates objects. While unlikely to truly OOM, it will produce a multi-thousand-element response array and a snapshot query with thousands of `IN (?)` placeholders — SQLite has a default `SQLITE_MAX_VARIABLE_NUMBER` of 999.

The snapshot query builds `weekPlaceholders` from `weekEntries.length`:
```sql
SELECT period, generated_at FROM reflect_snapshots
WHERE period IN (?, ?, ?, ... /* N+1 placeholders */) AND project_id = ?
```
If N > 998, this query will **fail** with a SQLite error.

**Fix:** Either cap `weekEntries` to a reasonable maximum (e.g., 200 weeks / ~4 years), or batch the snapshot query. The simpler fix is a safety cap on the loop.

---

### SUGGESTION

**2. The `- 86400000` buffer in the while condition is fragile**
```js
while (weekMondayMs >= earliestMondayMs - 86400000)
```
The comment says "small buffer for float precision," but these are integer millisecond values — there is no floating-point imprecision risk. Both values come from `Date.getTime()` which returns integers. The 1-day buffer means you might generate one extra week beyond the earliest session. This is harmless but the comment is misleading. Remove the buffer or change the comment to clarify it's intentional (e.g., "include the earliest week even if rounding lands just past its Monday").

**3. SQLite `date()` with `'weekday 1', '-7 days'` edge case**
The GROUP BY expression `date(s.started_at, 'weekday 1', '-7 days')` computes the preceding Monday. However, SQLite's `'weekday N'` modifier finds the **next** occurrence of that weekday (or same day if already that weekday). So for a Monday session, `'weekday 1'` returns the same Monday, then `'-7 days'` subtracts a week — yielding the **previous** Monday, not the current one. This is a known SQLite gotcha.

Verify: a session on Monday 2026-03-09 would get `date('2026-03-09', 'weekday 1')` = `2026-03-09` (already Monday), then `-7 days` = `2026-03-02`. That session would be counted in the week starting March 2, not March 9. This is a **logic bug** — sessions on Mondays are bucketed into the wrong week.

**Recommended fix:** Use `date(s.started_at, 'weekday 0', '-6 days')` instead (find next Sunday, subtract 6 = same week's Monday), or use a simpler expression like:
```sql
date(s.started_at, '-' || ((CAST(strftime('%w', s.started_at) AS INTEGER) + 6) % 7) || ' days')
```

If `currentWeek` happens to be one of the first 7 weeks, it appears twice (once from slice, once from find). The `filter(Boolean)` won't deduplicate — it only removes `undefined`. You'd render a duplicate dot with the same `key`, which React warns about.

**Fix:** Check for deduplication:

const visible = tail && !weeks.slice(0, 7).some(w => w.week === tail.week)

With 200+ weeks of history, the `IN (?, ?, ..., ?)` placeholder list for `reflect_snapshots` becomes large. ## Wild Card Review — PR #184: Data-Driven Week Navigation

### BLOCKING

**1. SQLite `date(..., 'weekday 1', '-7 days')` does NOT compute ISO Monday correctly**

The GROUP BY expression `date(s.started_at, 'weekday 1', '-7 days')` is wrong for sessions that start on a Monday. SQLite's `'weekday N'` modifier advances to the **next** occurrence of that weekday, or stays on the same day if it already matches. So for a Monday input:
- `'weekday 1'` = same Monday (no change)
- `'-7 days'` = previous Monday (one week too early!)

For a Tuesday input: `'weekday 1'` advances to next Monday, then `-7 days` goes back = correct (the preceding Monday). But Monday itself is off by one week.

This means **all sessions that start on a Monday will be bucketed into the previous week's count**. This is a data correctness bug.

**Fix:** Use `date(s.started_at, 'weekday 4', '-3 days')` instead — the Thursday trick. `'weekday 4'` finds the Thursday of the ISO week (advances to next Thursday, or stays if already Thursday), then `-3 days` gives the Monday. This handles all days correctly because the Thursday-to-Monday offset is always -3 within the same ISO week. Alternatively, handle the Monday edge case explicitly.

---

### SUGGESTION

**2. Unbounded week generation for users with very old sessions**

A user who imported Claude Code sessions from 2023 would generate ~150+ `WeekEntry` objects. For each, `formatIsoWeek` and `parseIsoWeek` are called, and a snapshot query uses `IN (...)` with 150+ placeholders. This isn't catastrophic for SQLite, but it's wasteful:
- Most of those weeks will have zero sessions
- The snapshot `IN` clause scales linearly with week count
- The `weekEntries` array is rebuilt on every request (no caching)

Consider: instead of generating ALL weeks from earliest to current, query distinct weeks that actually have sessions and fill gaps only in the recent window (e.g., last 12 weeks). The arrow navigation only visits weeks with sessions anyway (`navigableWeeks` filters on `sessionCount > 0`), so empty weeks in the middle serve no purpose.

**3. Float precision buffer is fragile**

```js

```

The comment says "float precision" but these are integer millisecond values — there's no float precision issue. The `- 86400000` (one full day) buffer is overly generous and could include an extra week before the earliest session. A buffer of `1` (one millisecond) would suffice if any buffer is needed at all. As-is, this is harmless but misleading.

- That there are weeks between the visible dots

The `formatIsoWeek` function handles this correctly via the Thursday trick, but the `date()` SQLite expression from Finding #1 would be especially problematic here — a Monday that falls on Dec 29 (2025) could be ISO week 2026-W01, and the off-by-one bug would misclassify it as 2025-W52.

The old code defined `projectCondition` after the week generation loop. The new code hoists it above the `earliestRow` query so it can be reused. This is a nice cleanup.

**7. GROUP BY approach is fundamentally better than CASE bucketing**

The old CASE bucketing approach generated N SQL parameters per week (2N total). With 8 weeks that was 16 params — manageable. With 150 weeks it would be 300 params in a single query. The GROUP BY approach is O(sessions-with-data) not O(weeks), which is the right direction. The implementation just needs the Monday computation fixed (Finding #1).

**8. The `filter(Boolean)` cast in the dot logic is safe but worth a comment**

```ts
[...weeks.slice(0, 7), weeks.find(w => w.week === currentWeek)].filter(Boolean) as typeof weeks
```

`weeks.find()` could return `undefined` if `currentWeek` isn't in the `weeks` array at all (e.g., during a race between navigation and data loading). The `filter(Boolean)` handles this, but the `as typeof weeks` cast hides the fact that the array could have fewer than 8 elements. This is fine in practice but a comment noting the defensive handling would help future readers.
