# /start-review — Triple-Layer Code Review Team

**PR**: 247

You are setting up a triple-layer code review for PR `247`. This can be used standalone or as part of a `/start-feature` team workflow. The review loops until all FIX NOW items are resolved — it does NOT end after a single pass.

---

## Step 1: Get PR Details

Fetch the PR details:

```bash
# Get the correct owner from git remote
git remote get-url origin | sed 's/.*[:/]\([^/]*\)\/[^/]*\.git/\1/'
```

Use `gh pr view 247` to get PR title, description, and diff stats.
Use `gh pr diff 247` to get the diff.

Determine the PR scope:
- Count files changed and lines changed per file
- Identify if it touches schema concerns (types.ts, SQLite schema, server API)
- Classify changed files into domains for specialist selection (Step 2)

---

## Step 2: Select Domain Specialists

Instead of fixed generalist reviewers, select 1-2 **domain specialists** based on what the PR actually changes. Each specialist combines deep domain expertise with general engineering awareness. See `docs/REVIEW-SPECIALISTS.md` for the full specialist registry and prompt templates.

### Domain Classification

Classify each changed file into a domain based on file patterns:

| Domain | File Pattern Triggers |
|--------|----------------------|
| **SQL/Database** | `server/src/routes/*.ts` (when containing SQL keywords), `cli/src/db/*.ts` |
| **React/Frontend** | `dashboard/src/**/*.tsx`, `dashboard/src/**/*.ts` |
| **Node/CLI** | `cli/src/commands/*.ts`, `cli/src/utils/*.ts`, `server/src/*.ts` (non-route, non-LLM) |
| **Parser/Provider** | `cli/src/providers/**/*.ts`, `cli/src/parser/*.ts` |
| **LLM** | `server/src/llm/**/*.ts` — handled by LLM Expert (separate, unchanged) |

Files that don't match any domain (e.g., `package.json`, `tsconfig.json`, `.gitignore`, `CLAUDE.md`) are ignored during classification.

### Selection Algorithm

1. For each domain, sum the lines changed across all files in that domain
2. **Primary specialist** = domain with the most lines changed
3. **Secondary specialist** = next domain IF it has >= 30% of total changed lines AND is a different domain
4. Cap at **2 domain specialists** (LLM Expert is a separate conditional reviewer)
5. If a PR changes ONLY non-domain files, use the **Node/CLI Specialist** as general-purpose

### LLM Expert (Conditional — Unchanged)

**Invoke LLM Expert if ANY of these apply:**
- PR touches `server/src/llm/` (prompts, providers, export-prompts)
- PR adds or modifies LLM API calls
- PR changes structured output schemas or SSE streaming
- PR modifies token budgets or model selection logic
- PR adds new LLM-powered features

**Skip LLM Expert if ALL of these apply:**
- No LLM code touched
- Pure UI, CLI, or schema changes
- Provider (source tool) implementations only

1. **Defensive `?? []` guards** on `response.decisions`, `response.learnings`, `response.findings`, `response.takeaways` — added from the CLI inline version. These protect against malformed LLM output where typed fields might be undefined at runtime. Safe, strictly more defensive.

### Issues Found

**🔴 FIX NOW: `saveAnalysisUsage` will clobber `session_message_count` on server re-analysis**

The old server `saveAnalysisUsage` intentionally **excluded** `session_message_count` from both the INSERT column list and the ON CONFLICT SET clause. The comment even documented this: *"preserves columns not in this write (e.g. session_message_count written by the CLI insights command)"*.

The new unified version in `cli/src/analysis/analysis-usage-db.ts` includes `session_message_count` in **both** INSERT and ON CONFLICT SET. When server callers invoke this (via re-export) without providing `session_message_count`, the `?? null` fallback writes `NULL`, **overwriting** a previously CLI-written value.

**Impact**: After a server-side re-analysis (backfill, dashboard "re-analyze" button), the `session_message_count` for that session will be set to NULL. The next CLI hook run will fail to detect the session as "already analyzed" (resume detection), causing unnecessary re-analysis.

**Fix**: Either:
- (A) Change the ON CONFLICT SET to conditionally preserve: `session_message_count = COALESCE(excluded.session_message_count, analysis_usage.session_message_count)` — this writes the new value when provided, preserves the old value when NULL.
- (B) Omit `session_message_count` from the ON CONFLICT SET clause entirely when the value is null (requires two prepared statements or dynamic SQL).

Option (A) is cleanest — single statement, correct semantics.

### File System Safety

**Settings.json read/write is reasonably safe.** The existing pattern — `existsSync` check, `readFileSync` + `JSON.parse` in a try/catch with fallback to empty object, `mkdirSync({ recursive: true })` before write — is adequate for a single-user CLI tool. No race conditions are realistically possible since only the user runs this command manually.

**Non-atomic write:** `fs.writeFileSync` is not atomic (no write-to-temp + rename pattern). If the process crashes mid-write, `settings.json` could be corrupted. However, the file is small (typically <1KB) and this matches the existing pattern that was already in production. Not a regression.

**Malformed JSON handling:** Covered — the existing try/catch around `JSON.parse` logs a warning and falls back to `{}`. This means a corrupted settings.json won't crash, but will silently discard existing settings. This is pre-existing behavior, not introduced by this PR.

### CLI UX & Flags

**Flag registration is correct.** In `cli/src/index.ts`, `--sync-only` and `--analysis-only` are registered via `.option()` and Commander.js will camelCase them to `syncOnly` and `analysisOnly`. The action handler maps them explicitly: `installHookCommand({ syncOnly: opts.syncOnly, analysisOnly: opts.analysisOnly })`.

**Output messaging is clear.** The default case explains both hooks with their purposes. The `--sync-only` and `--analysis-only` cases show targeted messages. The "already installed" messages are per-hook-type, which is good UX.

**"How it works" section after install** conditionally shows relevant info based on which hooks were installed. Good.

### Issues Found

**🔴 FIX NOW — No mutual exclusion for `--sync-only` + `--analysis-only`**

If a user passes both `--sync-only` and `--analysis-only`, the logic resolves to:
```
installSync = !analysisOnly = false
installAnalysis = !syncOnly = false
```
Neither hook gets installed. The function hits the `!syncInstalled && !analysisInstalled` guard and prints "Code Insights hooks already installed" with a suggestion to uninstall first — which is misleading because nothing was installed.

This is a logic bug. Options:
1. Add a check at the top: if both flags are set, print an error and return.
2. Treat both flags as equivalent to default (install both).

Option 1 is cleaner — these flags are intentionally mutually exclusive.

**🟡 SUGGESTION — `process.argv[1]` for CLI path**

The hook command is constructed as `node ${cliPath} sync -q` where `cliPath = process.argv[1]`. This is pre-existing behavior (not introduced by this PR), but worth noting:

- When installed via `npm link`, `argv[1]` is the symlinked path (e.g., `/usr/local/lib/node_modules/@code-insights/cli/dist/index.js`) — works fine.
- When run via `npx`, `argv[1]` points to a cache path that may change between runs — the hook could break after cache cleanup.
- When installed globally via `npm install -g`, `argv[1]` is stable — works fine.

This is a pre-existing concern, not a regression. But the new `SessionEnd` hook amplifies the blast radius since a broken path now affects two hooks instead of one.

**🟡 SUGGESTION — Missing `SessionEnd` in `ClaudeSettings` interface (in diff, not in Read)**

The diff adds `SessionEnd?: HookConfig[];` to the `ClaudeSettings` interface — good. But the Read tool showed the old file without it. Confirming from the diff that this is included. Correct.

**🔵 NOTE — Duplicate detection uses `includes('code-insights')`**

The `hookAlreadyInstalled` helper checks if any hook command string `.includes('code-insights')`. This is a reasonable heuristic but would false-positive on a hypothetical third-party tool whose command path happens to contain "code-insights". Low risk in practice — the user's `~/.claude/settings.json` is unlikely to have such a collision.

**🔵 NOTE — Test safety concern: tests operate on real `~/.claude/settings.json`**

The tests back up and restore `~/.claude/settings.json` via `beforeEach`/`afterEach`. This is functional but risky:
- If a test crashes before `afterEach` runs, the real settings file could be corrupted or deleted.

The backup/restore pattern mitigates this, and `vi.resetModules()` ensures fresh imports. But a more robust approach would be to mock `os.homedir()` to point to `tmpDir`. This is a pre-existing pattern concern if other test files do the same thing.

# /release — Automated Release Workflow

**Arguments**: minor

You are executing the release workflow for `@code-insights/cli`. Parse `minor` to extract:
- **type** (required): `patch`, `minor`, or `major`
- **description** (optional): A one-liner for the release title

If type is missing or not one of `patch`/`minor`/`major`, ask the user to provide it.

---

## Step 1: Pre-flight Checks

Run ALL of these checks. If any fail, STOP and tell the user what to fix.

```bash
# Must be on master
git branch --show-current  # Must output "master"

# Must have clean working tree
git status --porcelain     # Must be empty

# Pull latest
git pull origin master
```

Read the current version from `cli/package.json` (the `"version"` field).

Compute the new version:
- `patch`: increment the third number (3.6.1 → 3.6.2)
- `minor`: increment the second number, reset third to 0 (3.6.1 → 3.7.0)
- `major`: increment the first number, reset second and third to 0 (3.6.1 → 4.0.0)

**Resume detection**: Run `npm view @code-insights/cli version` to get the currently published version. If the version in `package.json` is ALREADY GREATER than the npm version, this is a **resumed release** — skip to Step 6.

---

## Step 2: Analyze Changes

Find the last git tag and analyze what changed:

```bash
# Find latest tag
git describe --tags --abbrev=0

# Commits since last tag
git log $(git describe --tags --abbrev=0)..HEAD --oneline

# Full diff since last tag
git diff $(git describe --tags --abbrev=0)..HEAD --stat
```

Read the full diff and commit messages. Then:

1. **Generate a CHANGELOG entry** in the existing format (see `cli/CHANGELOG.md` for style). Group changes under `### Added`, `### Changed`, `### Fixed`, `### Improved` as appropriate. Match the existing tone — concise but descriptive, with bold lead-ins.

2. **Generate a release title** — a short phrase summarizing the release (e.g., "LLM-Powered Export Page", "PostHog custom domain"). Use the user's description if provided, otherwise derive from the changes.

3. **For minor/major releases**: Assess which docs need updating:
   - `docs/ROADMAP.md` — if new milestones were completed
   - `docs/PRODUCT.md` — if product capabilities changed
   - `docs/VISION.md` — for major releases, if the vision expanded
   - `README.md` — if user-facing features or setup changed

---

```
  {- docs/ROADMAP.md (if applicable)}
  {- docs/PRODUCT.md (if applicable)}
  {- docs/VISION.md (if applicable)}

Approve, or tell me what to change?
```

Wait for user approval. If they provide edits, incorporate them.

After approval, make the changes:

1. **`cli/package.json`** — Update the `"version"` field to the new version
2. **`cli/CHANGELOG.md`** — Prepend the new entry at the top (after the header, before the previous version entry). Format: `## [X.Y.Z] - YYYY-MM-DD`
3. **Docs** (if approved at Gate 1) — Make the specific updates discussed

Run the full build and test suite:

## TA Design Review: SessionEnd Hook Cancellation Fix

### Root Cause Confirmation

The diagnosis is correct. `execFileSync('claude', ...)` in `ClaudeNativeRunner` spawns `claude -p` as a **child process of the hook subprocess**. Claude Code's hook manager owns that process tree and terminates it on completion/timeout. Since `claude -p` itself starts a Claude Code session, the hook system kills it. This is a process-tree containment problem, not a timing issue.

### Answers to Your Questions

**1. Process detachment strategy: `nohup ... &` vs `setsid` vs Node `spawn({ detached })`**

Recommendation: **Two-phase approach inside the Node code** (not at the shell level).

- `setsid` is Linux-only (not available on macOS without installing GNU coreutils). Rejected.
- `nohup ... &` at the shell level has the stdin piping problem you identified (question 2). Rejected as the sole approach.
- Node.js `child_process.spawn` with `{ detached: true, stdio: 'ignore' }` combined with `unref()` is the correct approach. It works identically on macOS and Linux. The spawned process gets a new process group (new session leader on Unix), so Claude Code's hook cleanup cannot reach it.

However, you do NOT want to detach `claude -p` itself. You want to detach a **wrapper process** that does: read stdin, sync the file, THEN run `claude -p`. This leads directly to question 3.

**2. Stdin piping concern**

Correct instinct. If you background at the shell level (`&`), stdin delivery is a race condition. The hook process writes stdin, but if the backgrounded child hasn't read it before the parent exits, the pipe breaks.

The fix is: **read stdin synchronously in the foreground hook process, then hand off the parsed data to the detached child.** Stdin must be consumed before any detachment happens.

**3. Architecture: Shell-level vs code-level detachment**

Recommendation: **Code-level detachment, split into two phases.**

The hook command stays as-is (no `nohup`, no `&` in the command string):
```
node /path/code-insights insights --hook --native -q
```

Inside `insightsCommand()` in `insights.ts`, when `opts.hook` is true:

- **Phase 1 (foreground, synchronous):** Read stdin, parse JSON, extract `session_id` and `transcript_path`, run `syncSingleFile()`. This must complete before the hook returns so the session is in the DB.
- **Phase 2 (detached):** Spawn a new Node process that runs the actual analysis: `node /path/code-insights insights <session_id> --native -q`. Use `child_process.spawn` with `{ detached: true, stdio: ['ignore', logFd, logFd] }` and call `child.unref()`. The parent process exits immediately, the hook completes, and the analysis runs in the background.

This is cleaner than shell-level detachment because:
- Stdin is consumed reliably in Phase 1
- The sync completes before the hook returns (data integrity)
- The detached process is a plain `insights <id> --native` call, no hook mode, no stdin dependency
- The hook command string stays simple (no shell escaping concerns with `nohup`/`&`)
- Cross-platform: works on macOS and Linux identically

**4. Error visibility / logging**

Yes, log to `~/.code-insights/hook-analysis.log`. Use append mode. Keep it simple: redirect stdout and stderr of the detached child to that file descriptor.

Open the log file with `fs.openSync(logPath, 'a')`, pass the fd as stdio for the spawned child. This gives you a persistent debug trail without any in-process logging infrastructure.

Add a note in `install-hook` output and/or `code-insights status` that tells users where to find the log. Do not add log rotation yet (YAGNI at current scale).

**5. Race condition: SessionEnd fires before Stop (sync) completes**

This is already handled by the proposed architecture. Phase 1 of the hook command calls `syncSingleFile()` with the `transcript_path` from stdin. This is independent of the Stop hook's full sync. The file is synced to SQLite before the detached analysis starts.

The only remaining race is: what if the session file is still being written when `SessionEnd` fires? This is a Claude Code guarantee question, not ours. `SessionEnd` should only fire after the session file is finalized. If you see truncated data, that is a Claude Code bug, not ours.

No additional mitigation needed here.

**6. Concerns and alternative evaluation**

The proposed approach (Option A with code-level detachment) is the right call. Specific concerns:

- **Option B (pending marker, process later):** Over-engineered. Requires a new table or file-based queue, a polling mechanism or secondary trigger, and adds latency. The analysis should run immediately while the session is fresh. Rejected.

- **Option C (`setsid`):** macOS incompatible. Rejected.

- **Concern: `execFileSync` in the detached process.** The detached child will call `runInsightsCommand()` which uses `ClaudeNativeRunner.execFileSync('claude', ...)`. This `claude -p` will be a grandchild of the detached process, but since the detached process is in its own process group (not under Claude Code's hook tree), the hook manager cannot reach it. This should work. If it does not (belt-and-suspenders), the detached child could also spawn `claude -p` with `{ detached: true }`, but I would not add that unless testing shows it is needed.

- **Concern: Multiple concurrent analyses.** If a user ends sessions rapidly, multiple detached analyses could run simultaneously. Each `claude -p` call is isolated (unique temp files with random suffixes already exist in `native-runner.ts`). SQLite writes use transactions. This is fine.

- **Concern: Hook timeout.** The 300s timeout in the hook config was sized for synchronous analysis. With detachment, Phase 1 (stdin + sync) should complete in under 5 seconds. You could reduce the timeout to 30s, but it is not strictly necessary since the process exits quickly anyway. Leave it at 300s for now to avoid unnecessary churn.

- **Filter at sync time:** Sessions with 2 or fewer messages AND zero tool calls should be skipped during sync. This matches the existing prune heuristic but applies it proactively. Add a `--include-trivial` flag for users who want everything.

- **Do NOT make this part of the v4.8.1 hotfix.** This is a separate quality-of-life improvement. Ship it in v4.9.0 or whatever follows. Do not bundle scope into the critical path fix.

- **Documentation:** Yes, mention `sync prune` in the install-hook output and in the "How it works" section. Low cost, immediately helpful for users who already have clutter.

---

### Question 5: Consolidating Hooks (Remove Stop Hook)

**Verdict: Strongly agree. Ship SessionEnd-only as the default.**

The Stop hook fires on every Claude Code response. For a 30-message session, that is 15+ syncs. Each sync is fast (sub-second), but it is still unnecessary write amplification and it means the database is constantly being written to while the user is working. That is the kind of thing that eventually causes a "why is my disk thrashing?" issue report.

SessionEnd gives you everything you need:
1. Sync the session file (the transcript is complete)
2. Analyze the session (analysis requires the full conversation anyway)

One hook, one trigger, one sync, one analysis. Simpler to explain, simpler to debug, fewer failure modes.

**Migration path:**
- `install-hook` defaults to SessionEnd only (sync + analysis in one command)
- Remove `--sync-only` and `--analysis-only` flags -- they add cognitive load for a split that no longer makes sense
- `uninstall-hook` still cleans up both Stop and SessionEnd hooks (for users who have the old config)
- Existing Stop hooks from v4.8.0 users: `install-hook` should detect and offer to migrate ("Found legacy Stop hook. Replacing with SessionEnd hook.")

**One concern:** Users who want real-time session data in the dashboard while working (seeing messages appear live) would lose that with SessionEnd-only. But this is an edge case, and those users can manually run `code-insights sync` if they want a mid-session update. Do not optimize the default for a power-user edge case.

### Question 6: Hook Feedback Messages (SessionStart/SessionEnd)

**Verdict: Cautious yes for SessionStart. No for SessionEnd.**

**SessionStart -- yes, one line:**
A single line at session start is low-friction confirmation that the hook is installed and working. This solves the "is it even doing anything?" anxiety that the current silent behavior creates. The message should be:

```
[Code Insights] Session will be analyzed on end.
```

That is it. No branding flourish, no emoji, no URL. One line that confirms the hook is active and sets the expectation ("on end" -- so they know it is not happening now).

**Why not SessionEnd:** The analysis is happening in the background (per the detach fix). Printing a message at SessionEnd that says "analyzing..." when the process is already detached and the terminal may be closing is unreliable and misleading. The user's next interaction with the results is the dashboard. Let the dashboard be the feedback.

**Implementation note:** This requires adding a SessionStart hook to `install-hook`. Keep it minimal -- just an echo/print command, no Node.js invocation. The lighter the SessionStart hook, the less risk of slowing down session startup.
