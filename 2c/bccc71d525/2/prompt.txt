```typescript
// llama.cpp provider — local inference via llama-server (OpenAI-compatible API)

import type { LLMClient, LLMMessage, LLMResponse, ChatOptions } from '../types.js';
import { flattenContent } from '../types.js';

const DEFAULT_LLAMACPP_URL = 'http://localhost:8080';

export function createLlamaCppClient(model: string, baseUrl?: string): LLMClient {
  const url = baseUrl || DEFAULT_LLAMACPP_URL;

  return {
    provider: 'llamacpp',
    model,

    async chat(messages: LLMMessage[], options?: ChatOptions): Promise&lt;LLMResponse&gt; {
      // ... fetch to `${url}/v1/chat/completions`
      // OpenAI-compatible request body: { model, messages, temperature: 0.7, max_tokens: 8192 }
      // flattenContent on each message's content
      // ECONNREFUSED detection same as ollama.ts but with llama-server specific message
      // Response parsing identical to openai.ts (choices[0].message.content, usage.prompt_tokens/completion_tokens)
    },

    estimateTokens(text: string): number {
      return Math.ceil(text.length / 4);
    },
  };
}

/**
 * Discover models loaded in llama-server via GET /v1/models.
 * Returns empty array if llama-server is not running.
 */
export async function discoverLlamaCppModels(
  baseUrl?: string
): Promise&lt;Array&lt;{ id: string; owned_by: string }&gt;&gt; {
  const url = baseUrl || DEFAULT_LLAMACPP_URL;
  try {
    const response = await fetch(`${url}/v1/models`, {
      signal: AbortSignal.timeout(3000),
    });
    if (!response.ok) return [];
    const data = await response.json() as { data?: Array&lt;{ id: string; owned_by: string }&gt; };
    return data.data || [];
  } catch {
    return [];
  }
}
```

Key design decisions for this file:
- **Connection error handling**: Mirror the Ollama pattern -- catch `ECONNREFUSED` and throw a user-friendly message: `"Cannot connect to llama-server at ${url} — is it running? Start it with: llama-server -m &lt;model.gguf&gt;"`
- **No auth header**: llama-server does not require API keys by default. No `Authorization` header sent.
- **JSON mode**: llama-server supports `response_format: { type: "json_object" }` in the request body. Include this to improve JSON reliability from quantized models. This is a key differentiator from the plain OpenAI client.
- **Model name flexibility**: llama-server's `/v1/models` returns the loaded model's name. If the user configured model `"custom"` or left it blank, the provider should still work because llama-server uses whatever model is loaded.

        response_format: { type: 'json_object' },
      }),
    });
    if (!response.ok) {
      const detail = await response.text().catch(() =&gt; '');
      throw new Error(`llama-server API error (HTTP ${response.status})${detail ? ` - ${detail}` : ''}`);
    }
    const data = await response.json() as {
      choices: Array&lt;{ message: { content: string } }&gt;;
      usage?: { prompt_tokens: number; completion_tokens: number };
    };
    return {
      content: data.choices[0]?.message?.content || '',
      usage: data.usage
        ? { inputTokens: data.usage.prompt_tokens, outputTokens: data.usage.completion_tokens }
        : undefined,
    };
  };
}

### Architectural Decisions

1. **llama.cpp as a separate provider vs. reusing OpenAI provider with custom baseUrl**: Separate provider is the right call. The OpenAI provider hardcodes `https://api.openai.com/v1/chat/completions` and always sends an `Authorization` header. llama.cpp has a different default URL, no auth, and benefits from `response_format: { type: "json_object" }` being always-on (the OpenAI provider does not send this). A separate provider is cleaner and follows the "no half-measures" philosophy.

2. **Model name for llama.cpp**: llama-server loads one model at a time. The "model" field in the config is somewhat cosmetic -- it identifies which model the user loaded. The `/v1/models` discovery endpoint confirms what is actually loaded. The provider should pass whatever model string the user configured in the request body, and llama-server will use the loaded model regardless.

3. **Gemma 4 on Gemini API**: If Google exposes Gemma 4 models through the Generative Language API (which they have done for Gemma 2), they would use the same Gemini provider with model IDs like `gemma-4-12b`. If not available via API, those entries can be removed from the Gemini provider's model list. Adding them now is the right call -- they are easy to remove if wrong.

4. **No streaming support**: The existing providers all use `stream: false`. The llama.cpp provider follows this pattern. Streaming is a separate concern tracked elsewhere.
